# テクニカル — 2026-05-31

**テーマ：** バッチ処理とオンラインバッチ  
**レベル：** N2/N3  
**分野：** システム設計 / ジョブ管理 / 業務処理  

---

## 📖 読解文（約800字）

業務システムでは、すべての処理をユーザーの操作に合わせてすぐ実行するわけではない。大量のデータをまとめて処理したり、他システムから受け取ったファイルを定期的に取り込んだりする場合は、バッチ処理がよく使われる。バッチ処理とは、決められた時刻や条件に基づいて、複数のデータを一括で処理する方式である。たとえば、夜間に売上データを集計する、毎朝CSVファイルを取り込む、月末に請求データを作成する、といった処理が代表的である。

一方、オンライン処理は、ユーザーや外部システムからのリクエストに対して、その場で結果を返す方式である。画面で検索ボタンを押して結果が表示される処理や、APIで注文情報を登録する処理はオンライン処理に近い。オンライン処理では応答時間が重要であり、数秒以内に結果を返すことが求められる。そのため、時間のかかる集計や大量更新をそのまま同期的に実行すると、タイムアウトやロック競合の原因になりやすい。

そこで実務では、オンラインバッチという考え方が使われることがある。オンラインバッチとは、ユーザー操作やAPIリクエストをきっかけに開始されるが、実際の重い処理はバックグラウンドのジョブとして実行する方式である。たとえば、画面から大量のCSVをアップロードすると、システムはすぐに「受付済み」と返し、その後で検証、登録、集計を非同期に進める。ユーザーは処理状況画面で「処理中」「完了」「エラー」などのステータスを確認する。

オンラインバッチを設計する際は、ジョブID、処理ステータス、リトライ、再実行、排他制御、エラー通知を明確にする必要がある。特に、同じファイルを二重に取り込まない仕組みや、途中で失敗した場合にどこから再開できるかは重要である。また、オンライン処理と同じDBを使う場合、バッチが長時間ロックを保持すると通常業務に影響する可能性がある。そのため、処理単位を小さく分ける、キューを使って負荷を平準化する、業務時間外に重い処理を寄せる、といった工夫が求められる。

バッチ処理は「古い方式」ではなく、業務データを安全に処理するための重要な設計手段である。オンラインバッチは、ユーザー体験と大量処理の安定性を両立させるための現実的な選択肢だと言える。

<details>
<summary>解答</summary>

3. 重い処理を同期的に実行すると、タイムアウトやロック競合の原因になりやすいため。

## 📐 文法・表現ポイント

### 1. 〜わけではない（N3/N2）
> すべての処理をユーザーの操作に合わせてすぐ実行するわけではない

「全部がそうではない」という部分否定を表す。  
例：バッチ処理は古い技術というわけではありません。

### 2. 〜に基づいて（N2）
> 決められた時刻や条件に基づいて、複数のデータを一括で処理する

「〜を根拠・基準として」という意味。  
例：業務カレンダーに基づいて、月末バッチを実行します。

### 3. 〜原因になりやすい（N3/N2）
> タイムアウトやロック競合の原因になりやすい

「そういう問題を起こす可能性が高い」という意味。  
例：大量更新を1トランザクションで実行すると、性能劣化の原因になりやすいです。

## 🎤 面接Q&A：バッチ関連

### Q1. バッチ処理とは何ですか？

**回答例：**
バッチ処理とは、一定の時刻や条件に基づいて、複数のデータをまとめて処理する方式です。たとえば、夜間に売上データを集計したり、CSVファイルを取り込んだり、月末に請求データを作成したりする処理が該当します。

**ベトナム語メモ：**
Batch là xử lý nhiều dữ liệu theo thời điểm/điều kiện nhất định, ví dụ import CSV, tổng hợp doanh thu, tạo billing cuối tháng.

### Q2. オンライン処理とバッチ処理の違いは何ですか？

**回答例：**
オンライン処理は、ユーザーや外部システムからのリクエストに対して、その場で結果を返す処理です。一方、バッチ処理は、大量データをまとめて処理することが多く、すぐに結果を返すよりも、安定して最後まで処理することが重要です。

**ベトナム語メモ：**
Online xử lý ngay và trả kết quả nhanh. Batch ưu tiên xử lý ổn định một khối dữ liệu lớn.

### Q3. オンラインバッチとは何ですか？

**回答例：**
オンラインバッチとは、ユーザー操作やAPIリクエストをきっかけに開始されるバッチ的な非同期処理です。画面やAPIには受付結果だけを返し、重い処理はバックグラウンドで実行します。処理状況はジョブIDやステータスで確認できるようにします。

**ベトナム語メモ：**
Online batch là xử lý batch/async được kích hoạt từ request online. Trả về nhận job trước, xử lý thật ở background.

### Q4. Azure Functions、Queue Storage、API Managementを使う構成はオンラインバッチですか？

**回答例：**
厳密には夜間バッチのような定時実行バッチではありませんが、オンライン処理を契機にして、重い処理をQueue経由で非同期に実行するため、オンラインバッチの一種として説明できます。特に、ジョブID、処理ステータス、リトライ、完了・エラー管理がある場合は、バッチ的な処理と言えます。

**ベトナム語メモ：**
Không phải scheduled batch truyền thống, nhưng có thể nói là một dạng online batch nếu request tạo job, queue xử lý async, có status/retry/done/error.

### Q5. その構成では、どのような処理フローになりますか？

**回答例：**
まず、API Managementでリクエストを受け付け、Azure Functionで入力チェックを行います。その後、DBにジョブ情報を登録し、Queue Storageにメッセージを追加します。APIはジョブIDと受付結果を返し、Queue TriggerのFunctionが実処理を行います。完了後はDBのステータスを完了またはエラーに更新します。

**ベトナム語メモ：**
APIM nhận request → Function validate → DB tạo job → Queue message → trả jobId → Queue Trigger xử lý → update DB done/error.

### Q6. なぜQueueを使いますか？

**回答例：**
重い処理をAPIの同期処理から切り離すためです。Queueを使うことで、APIの応答時間を短くし、処理量が増えた場合もバックグラウンドで順番に処理できます。また、リトライや負荷平準化もしやすくなります。

**ベトナム語メモ：**
Queue giúp tách xử lý nặng khỏi API, trả response nhanh, xử lý nền, dễ retry và san bằng tải.

### Q7. エラーが発生した場合はどうしますか？

**回答例：**
エラーが発生した場合は、ジョブのステータスをエラーに更新し、エラー内容をログに残します。再実行できる処理であれば、対象データやジョブIDを指定して再実行できるようにします。また、同じデータを二重に処理しないように、冪等性も考慮します。

**ベトナム語メモ：**
Khi lỗi: update status error, ghi log, cho retry theo jobId, và đảm bảo idempotency để tránh xử lý trùng.

### Q8. バッチ処理で注意する点は何ですか？

**回答例：**
主な注意点は、二重実行、途中失敗、DBロック、処理時間、リトライ、監視です。大量データを一度に処理すると負荷が高くなるため、処理単位を小さく分けたり、進捗を記録したりすることが重要です。

**ベトナム語メモ：**
Cần chú ý duplicate, fail giữa chừng, DB lock, thời gian xử lý, retry, monitoring.

### Q9. 面接で「バッチ経験がありますか」と聞かれたら、どう答えますか？

**回答例：**
はい、オンライン処理をきっかけにした非同期バッチの設計・実装経験があります。APIでリクエストを受け付け、ジョブ情報をDBに登録し、Queue経由でAzure Functionがバックグラウンド処理を行う構成です。処理状況、リトライ、エラー管理、二重実行防止を考慮しました。

**ベトナム語メモ：**
Trả lời có kinh nghiệm online-triggered async batch: API nhận, DB lưu job, Queue gọi Function xử lý, có status/retry/error/duplicate prevention.
