# テクニカル — SQLチューニングの実務ポイント

**テーマ：** SQLチューニングの実務ポイント

**レベル：** N2/N3

**分野：** データベース / パフォーマンス改善 / 障害調査

---

## 📖 読解文（約1000字）

SQLチューニングでは、単にSQLを書き換えるだけでなく、どこで時間がかかっているのかを計測し、原因を切り分けた上で改善する必要がある。業務システムでは、画面の一覧検索、帳票出力、夜間バッチ、APIの参照処理など、多くの場所でSQLが実行される。データ件数が少ない開発環境では問題が見えなくても、本番環境でデータが増えると、応答時間の悪化やタイムアウトが発生することがある。

調査の第一歩は、遅い処理を特定することである。アプリケーションログ、APM、DBのスロークエリログなどを確認し、どのSQLが何秒かかっているのか、何回実行されているのか、いつ発生しているのかを整理する。1回のSQLが遅い場合もあれば、1回は短くても同じSQLが大量に実行されて全体が遅くなる場合もある。そのため、SQL単体の実行時間だけでなく、実行回数と呼び出し元も確認する。

SQL単体の性能を確認する際に重要なのが実行計画である。実行計画を見ると、DBがどのテーブルから読み始め、どのインデックスを使い、どの結合方法で処理しているかが分かる。たとえば、対象データが少ないはずなのに全表走査になっている場合、検索条件に合ったインデックスがない、またはインデックスが使われにくい書き方になっている可能性がある。`WHERE`句で関数を使ってカラムを加工したり、前方一致ではない`LIKE`検索を多用したりすると、インデックスが効きにくくなることがある。

インデックスは検索を速くする有効な手段だが、増やせばよいというものではない。検索条件、結合条件、並び替え条件を確認し、実際によく使われる項目に絞って設計する。複合インデックスを作る場合は、列の順番も重要である。一方で、インデックスを増やしすぎると、登録や更新のたびにインデックスも更新されるため、更新性能やディスク使用量に影響する。改善前後で実行時間と実行計画を比較し、効果を確認することが必要である。

アプリケーション側でよく起きる問題として、N+1問題がある。たとえば注文一覧を取得した後、各注文ごとに顧客情報や明細情報を1件ずつ取得すると、注文が100件ある場合に追加で100回以上SQLが実行される。開発環境では気付きにくいが、件数が増えると急に遅くなる。N+1を防ぐには、必要なデータをJOINや一括取得でまとめて取得する、ORMの eager loading を使う、画面に本当に必要な項目だけ取得する、といった対策を検討する。

遅いSQLの原因は、検索だけではない。更新処理では、ロック待ちが原因で処理が遅くなることもある。長いトランザクション、不要に広い更新条件、同じレコードを複数処理が同時に更新する設計などは、ロック競合を起こしやすい。ロックが疑われる場合は、待機中のセッション、実行中のSQL、トランザクションの開始時刻を確認し、どの処理がどの処理を待たせているのかを調べる。

良いSQLチューニングとは、勘でSQLを書き換えることではなく、計測、実行計画の確認、原因の切り分け、改善、再計測を繰り返すことである。面接で説明する場合も、「遅いSQLを見つけた」「インデックスを追加した」だけで終わらせず、どのログを見て、実行計画で何を確認し、どのような改善を行い、どの程度改善したかまで話せると、実務経験として伝わりやすい。

1. SQLチューニングで、SQLを書き換える前に確認すべきことは何ですか？
2. 実行計画を見ると、どのような情報が分かりますか？
3. インデックスを増やしすぎると、どのような問題が起きる可能性がありますか？
4. N+1問題とは何ですか？
5. ロック待ちが疑われる場合、どのような情報を確認しますか？

<details>
<summary>解答</summary>

1. どのSQLが遅いのか、何秒かかっているのか、何回実行されているのか、どこから呼ばれているのかを確認する。
2. DBがどのテーブルから読み始め、どのインデックスを使い、どの結合方法で処理しているかが分かる。
3. 登録や更新のたびにインデックスも更新されるため、更新性能やディスク使用量に影響する可能性がある。
4. 一覧取得後に各行ごとに追加SQLを実行し、件数が増えるほどSQL実行回数が増えて遅くなる問題。
5. 待機中のセッション、実行中のSQL、トランザクションの開始時刻、どの処理がどの処理を待たせているかを確認する。

</details>

## 📐 文法・表現ポイント

### 1. 〜た上で（N2）
> 原因を切り分け**た上で**改善する必要がある

「先にAをして、その結果を前提にBをする」という意味。
例：実行計画を確認した上で、インデックスを追加するか判断します。

### 2. 〜だけでなく（N3/N2）
> SQL単体の実行時間**だけでなく**、実行回数も確認する

「AだけではなくBも」という意味。調査範囲を広げて説明するときに便利。
例：SQLだけでなく、アプリケーション側の呼び出し回数も確認します。

### 3. 〜可能性がある（N3/N2）
> インデックスが使われにくい書き方になっている**可能性がある**

断定せずに原因候補を説明する表現。
例：ロック待ちが原因で処理が遅くなっている可能性があります。

## 🎤 面接Q&A：SQLチューニング

### Q1. SQLチューニングでは、最初に何を確認しますか？

**回答例：**
まず、どの処理が遅いのかをログや監視ツールで確認します。その上で、対象SQL、実行時間、実行回数、呼び出し元、発生時間帯を整理します。1回のSQLが遅いのか、同じSQLが大量に実行されているのかを切り分けることが重要です。

**ベトナム語メモ：**
Đầu tiên xác định xử lý/SQL nào chậm, thời gian chạy, số lần chạy, nơi gọi và thời điểm phát sinh.

### Q2. 実行計画では何を見ますか？

**回答例：**
テーブルのアクセス順、インデックスの利用有無、結合方法、読み取る行数の見積もりなどを確認します。想定より多くの行を読んでいる場合や全表走査になっている場合は、検索条件やインデックス設計を見直します。

**ベトナム語メモ：**
Xem thứ tự access table, index có dùng không, join method, số dòng ước tính.

### Q3. インデックスを追加すれば必ず速くなりますか？

**回答例：**
必ずしも速くなるとは限りません。検索条件や結合条件に合っていないインデックスは効果が小さいです。また、インデックスを増やしすぎると登録や更新が遅くなるため、実行計画とデータ件数を確認した上で判断します。

**ベトナム語メモ：**
Không phải cứ thêm index là nhanh. Cần đúng điều kiện tìm kiếm/join, và quá nhiều index làm update chậm.

### Q4. N+1問題をどのように説明しますか？

**回答例：**
N+1問題とは、一覧を1回取得した後、各行に対して追加のSQLを実行してしまう問題です。件数が増えるほどSQL実行回数が増え、画面表示が遅くなります。対策として、JOIN、一括取得、ORMの eager loading などを検討します。

**ベトナム語メモ：**
N+1 là lấy list xong lại query từng dòng, làm số query tăng theo số record.

### Q5. ロック待ちが原因で遅い場合、どう調査しますか？

**回答例：**
DBのセッション情報を確認し、待機中の処理、実行中のSQL、トランザクションの開始時刻を見ます。どの処理がロックを保持し、どの処理が待たされているかを確認します。その上で、トランザクション範囲や更新条件を見直します。

**ベトナム語メモ：**
Kiểm tra session DB, SQL đang chạy, transaction start time, ai giữ lock và ai đang chờ.

### Q6. 遅いSQLを改善した経験を面接でどう話しますか？

**回答例：**
遅い画面について、まずログで対象SQLと実行時間を確認しました。次に実行計画を確認し、検索条件に対して適切なインデックスが使われていないことを把握しました。SQLの条件とインデックスを見直し、修正後に再計測して、応答時間が改善したことを確認しました。

**ベトナム語メモ：**
Nói theo flow: log -> SQL/time -> execution plan -> nguyên nhân -> sửa SQL/index -> đo lại.

### Q7. SQLチューニングで注意していることは何ですか？

**回答例：**
勘で修正せず、必ず計測結果と実行計画を確認することです。また、検索性能だけでなく、更新性能やロックへの影響も確認します。改善後は、実行時間、実行回数、エラー有無を再確認し、副作用がないかを見るようにしています。

**ベトナム語メモ：**
Không sửa theo cảm giác. Cần đo, xem plan, kiểm tra ảnh hưởng tới update/lock và đo lại sau khi sửa.

## 🧩 今日の練習

1. Trước tiên, tôi kiểm tra SQL chậm bằng log.
2. Tôi xem execution plan để kiểm tra index có được sử dụng không.
3. Nếu có N+1, số lần chạy SQL sẽ tăng theo số record.
4. Index giúp search nhanh hơn, nhưng quá nhiều index làm update chậm.
5. Sau khi sửa, cần đo lại để xác nhận hiệu quả cải thiện.

<summary>解答例</summary>

1. まず、ログで遅いSQLを確認します。
2. 実行計画を見て、インデックスが使われているか確認します。
3. N+1が発生すると、レコード件数に応じてSQLの実行回数が増えます。
4. インデックスを使うと検索は速くなりますが、増やしすぎると更新が遅くなります。
5. 修正後に再計測し、改善効果を確認する必要があります。

</details>
