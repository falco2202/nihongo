# テクニカル — 障害対応・ログ調査の実務ポイント

**テーマ：** 障害対応・ログ調査の実務ポイント

**レベル：** N2/N3

**分野：** 運用 / インシデント対応 / ログ調査 / 再発防止

---

## 📖 読解文（約1000字）

システム障害が発生したとき、最初に重要なのは、すぐに原因を断定することではなく、影響範囲を確認し、サービスへの影響を最小限に抑えることである。たとえば、ユーザーがログインできない、注文登録が失敗する、夜間バッチが途中で停止する、APIの応答時間が急に悪化するなど、障害の内容によって確認すべき情報は変わる。まずは、いつから発生しているのか、どの機能で起きているのか、全ユーザーに影響しているのか、一部のユーザーだけなのかを整理する。

一次対応では、復旧を優先する。監視アラート、アプリケーションログ、サーバーログ、DBログ、リリース履歴などを確認し、直近の変更や異常なエラーを探す。もし直前にリリースが行われていた場合は、リリース内容と障害の発生時刻を照合する。新しいコード、設定変更、DBマイグレーション、外部サービスの仕様変更などが原因候補になることがある。ただし、時刻が近いだけで原因と決めつけず、ログや再現結果で確認することが必要である。

ログ調査では、エラーメッセージだけでなく、リクエストID、ユーザーID、処理ID、ジョブID、発生時刻、ステータスコード、例外の種類を確認する。複数のサービスが連携している場合は、同じリクエストIDやトレースIDを使って、どのサービスで失敗したかを追跡する。たとえば、画面では「登録に失敗しました」としか表示されていなくても、バックエンドのログを見ると、DBの制約違反、外部APIのタイムアウト、権限エラー、データ不整合など、具体的な原因候補が分かることがある。

原因調査では、仮説を立てて一つずつ切り分ける。アプリケーションの問題なのか、DBの問題なのか、ネットワークの問題なのか、外部サービスの問題なのかを確認する。再現条件を整理し、同じ入力や同じユーザーで再現するか、特定の時間帯だけで発生するか、件数が多い場合だけ失敗するかを見る。ログだけで分からない場合は、メトリクス、APM、DBセッション、キューの滞留状況なども確認する。

暫定対応と恒久対応を分けて考えることも重要である。暫定対応とは、サービスを早く復旧させるための一時的な対応である。たとえば、問題のあるリリースをロールバックする、失敗したジョブを再実行する、キューに残ったデータを手動で処理する、設定を一時的に戻す、といった対応がある。一方、恒久対応では、同じ障害が再発しないように、根本原因を取り除く。コード修正、テスト追加、監視強化、リリース手順の見直し、運用手順書の更新などが必要になる。

障害対応の最後には、報告と再発防止が欠かせない。報告では、発生日時、影響範囲、原因、対応内容、復旧時刻、再発防止策を分かりやすくまとめる。上司や顧客に説明する場合は、技術的な詳細だけでなく、業務影響と今後の対策を明確に伝えることが求められる。面接で障害対応を説明する場合も、「ログを見ました」だけで終わらせず、影響確認、一次対応、原因の切り分け、復旧、再発防止、報告までの流れを話せると、実務経験として伝わりやすい。

<details>
<summary>解答</summary>

1. 発生時刻、対象機能、影響範囲、全ユーザーか一部ユーザーか、サービスへの影響などを確認する。
2. 例：リクエストID、ユーザーID、処理ID、ジョブID、発生時刻、ステータスコード、例外の種類。
3. アプリ、DB、ネットワーク、外部サービスなど、どこに問題があるかを仮説ごとに確認して絞り込む作業。
4. 暫定対応は早く復旧するための一時対応。恒久対応は根本原因を取り除き、再発を防ぐ対応。
5. 例：発生日時、影響範囲、原因、対応内容、復旧時刻、再発防止策。

## 📐 文法・表現ポイント

### 1. 〜ことではなく、〜ことである（N2）
> 最初に重要なのは、すぐに原因を断定することではなく、影響範囲を確認することである

対比しながら「本当に重要なこと」を説明する表現。
例：障害対応で大切なのは、責任を探すことではなく、事実を整理することです。

### 2. 〜だけでなく（N3/N2）
> エラーメッセージだけでなく、リクエストIDや発生時刻も確認する

調査項目を広げるときに使いやすい表現。
例：アプリログだけでなく、DBログと監視メトリクスも確認します。

### 3. 〜と決めつけず（N2）
> 時刻が近いだけで原因と決めつけず、ログで確認する

思い込みを避けて説明するときの表現。
例：外部サービスが原因と決めつけず、まず自分たちのログも確認します。

## 🎤 面接Q&A：障害対応・ログ調査

### Q1. 障害が発生したとき、最初に何をしますか？

**回答例：**
まず、影響範囲と発生時刻を確認します。どの機能で発生しているのか、全ユーザーに影響しているのか、一部ユーザーだけなのかを整理します。その上で、監視アラートやログを確認し、復旧を優先して一次対応を行います。

**ベトナム語メモ：**
Đầu tiên xác nhận phạm vi ảnh hưởng, thời điểm phát sinh, chức năng bị ảnh hưởng, rồi ưu tiên phục hồi.

### Q2. ログ調査では、どのような情報を確認しますか？

**回答例：**
エラーメッセージ、発生時刻、ステータスコード、例外の種類、リクエストID、ユーザーID、処理IDなどを確認します。複数サービスが連携している場合は、トレースIDを使って、どのサービスで失敗したかを追跡します。

**ベトナム語メモ：**
Xem error message, time, status code, exception, request ID, user ID, process ID, trace ID.

### Q3. 原因を切り分けるとき、どのように進めますか？

**回答例：**
まず仮説を立てます。アプリの問題なのか、DBの問題なのか、ネットワークや外部APIの問題なのかを分けて確認します。再現条件、直近のリリース、メトリクス、DBセッションなどを確認し、原因候補を一つずつ絞り込みます。

**ベトナム語メモ：**
Đặt giả thuyết và khoanh vùng theo app, DB, network, external API; kiểm tra release, metrics, session DB.

### Q4. 暫定対応と恒久対応の違いは何ですか？

**回答例：**
暫定対応は、サービスを早く復旧させるための一時的な対応です。たとえば、ロールバックやジョブの再実行があります。恒久対応は、根本原因を取り除き、再発を防ぐ対応です。コード修正、テスト追加、監視強化、手順書の見直しなどを行います。

**ベトナム語メモ：**
Temporary workaround để khôi phục nhanh. Permanent fix để loại bỏ root cause và tránh tái phát.

### Q5. 障害報告では何を伝えますか？

**回答例：**
発生日時、影響範囲、原因、対応内容、復旧時刻、再発防止策を伝えます。顧客や上司に説明する場合は、技術的な詳細だけでなく、業務影響と今後の対策を分かりやすく説明します。

**ベトナム語メモ：**
Báo cáo thời gian, phạm vi ảnh hưởng, nguyên nhân, xử lý, thời điểm phục hồi, biện pháp phòng ngừa.

### Q6. 障害対応の経験を面接でどう説明しますか？

**回答例：**
障害対応では、まず監視アラートと問い合わせ内容から影響範囲を確認しました。次に、アプリケーションログとDBログを確認し、リクエストIDで関連ログを追跡しました。原因候補を切り分けた上で暫定対応を行い、復旧後に根本原因を整理して、再発防止としてコード修正と監視項目の追加を行いました。

**ベトナム語メモ：**
Nói theo flow thực tế: alert/user report -> impact -> log/DB log -> request ID -> workaround -> root cause -> prevention.

### Q7. ログだけで原因が分からない場合はどうしますか？

**回答例：**
ログだけで判断できない場合は、メトリクス、APM、DBセッション、キューの滞留状況、外部サービスのステータスなども確認します。また、同じ条件で再現するかを確認し、必要であれば関係者にリリース内容や運用作業の有無を確認します。

**ベトナム語メモ：**
Nếu log chưa đủ, xem metrics, APM, DB session, queue backlog, external service status và hỏi release/operation gần đây.
