# テクニカル — 非同期処理・キューの実務ポイント

**テーマ：** 非同期処理・キューの実務ポイント

**レベル：** N2/N3

**分野：** 非同期処理 / メッセージキュー / リトライ / デッドレター / 負荷平準化

---

## 📖 読解文（約1000字）

非同期処理とは、リクエストを受け付けた後、その場ですべての処理を完了させず、時間のかかる処理をバックグラウンドで実行する方式である。たとえば、ユーザーが画面から大きなCSVファイルをアップロードした場合、システムがすぐに全件を検証して登録しようとすると、画面の応答が遅くなり、タイムアウトする可能性がある。このような場合は、まず「受付済み」と返し、実際の検証や登録は別の処理で進める設計にする。

非同期処理でよく使われるのがキューである。キューは、処理してほしい仕事を一時的にためておく入れ物のような役割を持つ。リクエストを受け付ける側は、メッセージをキューに登録する。この登録する側をプロデューサーと呼ぶことがある。一方、キューからメッセージを取り出して実際の処理を行う側をコンシューマーまたはワーカーと呼ぶ。プロデューサーとコンシューマーを分けることで、画面やAPIの応答を速くし、重い処理を後ろで安全に実行できる。

キューを使う大きなメリットの一つは、負荷平準化である。短時間に大量のリクエストが来た場合でも、すべてを同時に処理しようとせず、キューにためて順番に処理できる。これにより、DBや外部APIへの負荷を急に増やさず、システム全体を安定させやすくなる。必要に応じてワーカーの数を増やせば、処理能力を上げることもできる。ただし、ワーカーを増やしすぎるとDBロックや外部APIの制限にぶつかることもあるため、処理件数、実行時間、エラー率を見ながら調整する必要がある。

非同期処理では、失敗時のリトライ設計が重要である。外部APIの一時的な障害やネットワークの瞬断で失敗した場合、少し時間を置いて再実行すれば成功することがある。そのため、最大リトライ回数、リトライ間隔、同じエラーを何回まで許容するかを決める。一方、入力データの形式が不正な場合や、必須項目が不足している場合は、何度リトライしても成功しない。このようなエラーを無限にリトライすると、キューが詰まり、他の処理にも影響する。

リトライしても成功しないメッセージは、デッドレターキューに移す設計にすることがある。デッドレターキューとは、通常の処理では完了できなかったメッセージを隔離しておくためのキューである。担当者は、デッドレターに入ったメッセージを確認し、原因を調査する。たとえば、データ形式の変更、マスタ未登録、外部サービスの仕様変更などが原因になっていることがある。調査後、データを修正して再投入するか、業務的に破棄するかを判断する。

キュー処理では、同じメッセージが複数回処理される可能性も考慮する必要がある。多くのキューサービスでは、メッセージを取り出した後、処理が完了するまで一時的に他のワーカーから見えない状態にする。この時間を可視性タイムアウトと呼ぶことがある。処理完了前にワーカーが落ちると、同じメッセージが再び見えるようになり、別のワーカーが処理する。そのため、処理は冪等に設計し、同じジョブIDやメッセージIDが来ても二重登録や二重送信が起きないようにする。

面接で非同期処理やキューを説明する場合は、「キューを使いました」だけではなく、なぜ同期処理ではなく非同期にしたのか、どの処理をキューに渡したのか、リトライとデッドレターをどう設計したのか、重複処理をどう防いだのか、負荷平準化にどう役立ったのかを話せるとよい。これらを説明できると、単なる実装経験ではなく、運用まで考えた設計経験として伝わりやすい。

## ❓ 読解問題

1. 非同期処理とは、どのような方式ですか？
2. キューにメッセージを登録する側と、取り出して処理する側を何と呼びますか？
3. キューを使うと、なぜ負荷平準化に役立ちますか？
4. リトライしても成功しないメッセージは、どこに移すことがありますか？
5. キュー処理で冪等性が必要になる理由を説明してください。

<details>
<summary>解答</summary>

1. リクエストを受け付けた後、その場ですべて完了させず、時間のかかる処理をバックグラウンドで実行する方式。
2. 登録する側をプロデューサー、取り出して処理する側をコンシューマーまたはワーカーと呼ぶ。
3. 大量リクエストを一度に処理せず、キューにためて順番に処理できるため、DBや外部APIへの急な負荷を抑えられるから。
4. デッドレターキューに移すことがある。
5. ワーカー停止や可視性タイムアウトなどにより、同じメッセージが複数回処理される可能性があるため。

</details>

## 📐 文法・表現ポイント

### 1. 〜ずに（N3）
> その場ですべての処理を完了させず、バックグラウンドで実行する

「〜しないで」という意味。技術説明では、避けたい処理を説明するときに使いやすい。
例：重い処理を同期的に実行せずに、キューへ渡します。

### 2. 〜ことで（N3/N2）
> プロデューサーとコンシューマーを分けることで、応答を速くできる

手段や理由を表す表現。
例：キューを使うことで、負荷を平準化できます。

### 3. 〜可能性がある（N3/N2）
> 同じメッセージが複数回処理される可能性がある

リスクや想定される事象を説明するときに使う表現。
例：外部APIが遅い場合、タイムアウトする可能性があります。

## 🎤 面接Q&A：非同期処理・キュー

### Q1. 非同期処理とは何ですか？

**回答例：**
非同期処理とは、リクエストを受け付けた後、その場ですべての処理を完了させず、時間のかかる処理をバックグラウンドで実行する方式です。画面やAPIにはまず受付結果を返し、実際の重い処理はワーカーが後で実行します。

**ベトナム語メモ：**
Nhận request trước, trả kết quả tiếp nhận, xử lý nặng chạy phía sau.

### Q2. なぜキューを使いますか？

**回答例：**
キューを使うことで、リクエストを受け付ける処理と、重い処理を実行する処理を分けられます。これにより、APIの応答を速くし、大量リクエストが来てもキューにためて順番に処理できます。また、ワーカー数を調整することで、処理能力をコントロールできます。

**ベトナム語メモ：**
Queue tách nhận request và xử lý nặng, giúp response nhanh và san bằng tải.

### Q3. リトライ設計では何を決めますか？

**回答例：**
最大リトライ回数、リトライ間隔、リトライ対象のエラー、リトライしないエラーを決めます。たとえば、外部APIの一時的なタイムアウトはリトライしますが、入力データの形式不正はリトライしても成功しないため、デッドレターに移す設計にします。

**ベトナム語メモ：**
Quyết định retry count, interval, lỗi nào retry, lỗi nào không retry.

### Q4. デッドレターキューとは何ですか？

**回答例：**
デッドレターキューとは、通常の処理で完了できなかったメッセージを隔離するためのキューです。最大リトライ回数を超えたメッセージや、処理不能なデータを移し、後で担当者が原因を確認します。原因を修正できる場合は再投入し、業務的に不要な場合は破棄します。

**ベトナム語メモ：**
DLQ chứa message lỗi không xử lý được sau retry, để điều tra và xử lý thủ công.

### Q5. キュー処理で重複処理を防ぐにはどうしますか？

**回答例：**
ジョブIDやメッセージIDを使って、すでに処理済みかどうかを確認します。また、DB側に一意制約を設定し、同じデータが二重登録されないようにします。キュー処理では同じメッセージが複数回届く可能性があるため、処理を冪等にすることが重要です。

**ベトナム語メモ：**
Dùng job ID/message ID/unique constraint; thiết kế idempotent vì message có thể được xử lý nhiều lần.

### Q6. 負荷平準化とは何ですか？

**回答例：**
負荷平準化とは、短時間に集中したリクエストを一度に処理せず、キューにためて少しずつ処理することです。これにより、DBや外部APIへの負荷を抑え、システムを安定させやすくなります。ワーカー数を増減することで、処理速度も調整できます。

**ベトナム語メモ：**
Load leveling là giữ request trong queue và xử lý dần để hệ thống ổn định.

### Q7. 面接で「非同期処理の経験」をどう説明しますか？

**回答例：**
非同期処理では、画面やAPIでリクエストを受け付けた後、重い処理をキューに登録する設計にしました。ワーカーがキューからメッセージを取り出して処理し、一時的なエラーはリトライしました。最大リトライ回数を超えた場合はデッドレターキューに移し、担当者が調査できるようにログとメッセージIDを残しました。また、重複処理を防ぐために、ジョブIDで処理済みチェックを行いました。

**ベトナム語メモ：**
Nói theo flow: request -> enqueue -> worker -> retry -> DLQ -> log/message ID -> idempotency.
