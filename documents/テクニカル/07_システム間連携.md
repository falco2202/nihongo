**レベル：** N2/N3

**分野：** API連携 / CSV連携 / ファイル取込 / リトライ / 重複防止

## 📖 読解文（約1000字）

システム間連携とは、複数のシステムがデータを受け渡しながら、一つの業務処理を完了させる仕組みである。たとえば、ECサイトで注文が登録された後、在庫管理システムに在庫引当を依頼し、決済システムに支払い処理を依頼し、配送システムに出荷指示を送るようなケースがある。利用者から見ると一つの注文処理に見えても、裏側では複数のシステムが連携している。

連携方式にはいくつかの種類がある。リアルタイム性が必要な場合はAPI連携を使うことが多い。API連携では、リクエストとレスポンスの項目、ステータスコード、タイムアウト、認証方式、エラー時の扱いを明確にする必要がある。たとえば、外部APIが一時的に遅くなった場合に、何秒でタイムアウトするか、何回リトライするか、リトライしても失敗した場合にどのようなエラーとして扱うかを決めておく。

一方、大量データをまとめて渡す場合や、相手システムがAPIを持っていない場合は、CSV連携やファイル取込を使うことがある。たとえば、基幹システムから毎朝CSVファイルを出力し、別システムがそのファイルを取り込んでマスタ情報を更新するような運用である。この場合は、ファイル名、文字コード、区切り文字、必須項目、日付形式、取込済みファイルの扱い、エラー行の扱いを事前に決める必要がある。

システム間連携で特に注意が必要なのは、失敗時の設計である。ネットワーク障害、相手システムのメンテナンス、データ不整合、認証エラー、ファイル形式の変更などにより、連携処理は失敗することがある。そのため、正常系だけでなく、異常系を具体的に設計することが重要である。エラーが発生した場合は、処理を中断するのか、一部だけスキップするのか、後で再実行するのか、担当者に通知するのかを決めておく。

リトライを設計するときは、重複処理を防ぐことも重要である。たとえば、注文登録APIへのリクエストがタイムアウトした場合、クライアント側から見ると失敗に見える。しかし、サーバー側ではすでに注文が登録されている可能性がある。この状態で単純に同じリクエストを再送すると、注文が二重登録されるリスクがある。対策として、リクエストIDや冪等性キーを使い、すでに処理済みかどうかを確認する。また、DBの一意制約や取込履歴テーブルを使って、同じデータを二回処理しないようにする。

CSV連携やファイル取込でも、重複防止の考え方は同じである。ファイル単位で取込履歴を管理し、同じファイル名や同じ連携IDのファイルを再度取り込まないようにする。行単位でエラーが発生する場合は、正常行だけを取り込むのか、ファイル全体をエラーにするのかを業務要件に合わせて決める。障害調査のためには、連携ID、ファイル名、取込日時、件数、エラー内容をログに残すことも必要である。

面接でシステム間連携を説明する場合は、「APIを呼びました」「CSVを取り込みました」だけでは不十分である。連携方式を選んだ理由、データ項目の設計、エラー処理、リトライ、重複防止、ログ出力、運用時の再実行方法まで説明できると、実務で考えるべきポイントを理解していることが伝わりやすい。

### 1. 〜場合は（N4/N3）

条件を説明するときに使う基本表現。
例：大量データをまとめて渡す場合は、CSV連携を検討します。

### 2. 〜ておく（N4/N3）
> エラー時の扱いを決めておく

将来のために、前もって準備する意味。
例：障害時に再実行できるように、取込履歴を残しておきます。

### 3. 〜だけでは不十分である（N2）
> 「CSVを取り込みました」だけでは不十分である

説明が足りないことを述べ、追加で必要な観点を示す表現。
例：正常系だけでは不十分で、異常系も設計する必要があります。

## 🎤 面接Q&A：システム間連携

### Q1. システム間連携とは何ですか？

**回答例：**
システム間連携とは、複数のシステムがデータを受け渡しながら、一つの業務処理を完了させる仕組みです。たとえば、注文システム、在庫管理システム、決済システム、配送システムが連携して、注文から出荷までの処理を進めます。

**ベトナム語メモ：**
Nhiều hệ thống trao đổi dữ liệu để hoàn thành một quy trình nghiệp vụ.

### Q2. API連携とCSV連携は、どのように使い分けますか？

**回答例：**
リアルタイム性が必要な場合や、すぐに結果を返したい場合はAPI連携を使います。一方、大量データをまとめて渡す場合や、相手システムがAPIを持っていない場合はCSV連携やファイル取込を使います。業務要件、データ量、処理タイミング、運用方法を見て方式を選びます。

**ベトナム語メモ：**
API cho realtime; CSV/file cho batch, dữ liệu lớn, hoặc hệ thống đối tác không có API.

### Q3. システム間連携で設計すべき異常系は何ですか？

**回答例：**
外部APIのタイムアウト、相手システムの停止、認証エラー、データ不整合、CSVの形式不正、必須項目不足などを考慮します。エラーが発生した場合に、処理を中断するのか、スキップするのか、リトライするのか、通知するのかを事前に決めます。

**ベトナム語メモ：**
Thiết kế timeout, downtime, auth error, data mismatch, CSV format error, missing required fields.

### Q4. リトライ設計で注意することは何ですか？

**回答例：**
一時的なエラーであればリトライは有効ですが、同じ処理が二重に実行されないように注意します。リクエストIDや冪等性キーを使って処理済みかどうかを確認し、DB側でも一意制約を設定して重複登録を防ぎます。また、何回リトライするか、間隔をどうするかも決めます。

**ベトナム語メモ：**
Retry phải đi cùng chống trùng: request ID, idempotency key, unique constraint, retry count/interval.

### Q5. CSVファイル取込で注意することは何ですか？

**回答例：**
ファイル名、文字コード、区切り文字、必須項目、日付形式、件数、エラー行の扱いを明確にします。また、同じファイルを二回取り込まないように、取込履歴を管理します。障害時に再実行できるよう、処理結果とエラー内容をログに残します。

**ベトナム語メモ：**
Chú ý tên file, encoding, delimiter, required fields, date format, error rows, import history.

### Q6. 面接で「システム間連携の経験」をどう説明しますか？

**回答例：**
システム間連携では、まず業務要件を確認し、リアルタイム連携が必要か、バッチ連携でよいかを整理しました。その上で、API連携またはCSV連携の方式を決め、連携項目、エラー処理、リトライ、重複防止、ログ出力を設計しました。運用時に再実行できるように、連携IDや取込履歴も管理しました。

**ベトナム語メモ：**
Nói theo flow: requirement -> choose API/CSV -> fields -> error handling -> retry -> duplicate prevention -> logs -> rerun operation.

### Q7. 重複防止はどのように実現しますか？

**回答例：**
API連携では、リクエストIDや冪等性キーを使い、同じリクエストが再送されても二重登録しないようにします。CSV連携では、取込履歴テーブルでファイル名や連携IDを管理し、同じファイルを再度取り込まないようにします。さらに、DBの一意制約でも重複を防ぎます。

**ベトナム語メモ：**
API dùng request ID/idempotency key; CSV dùng import history; DB dùng unique constraint.
