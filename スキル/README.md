## 📖 読解文（約500字）

日本のIT職場でのコードレビューは、単に技術的な問題を指摘するだけでなく、チームのコミュニケーション文化を反映する場でもある。特に、レビューコメントの書き方には独特の配慮が求められることが多い。直接的な否定表現を避け、提案形式や疑問形を使うことで、受け取る側が萎縮しないよう工夫されている。

たとえば「この書き方は間違っています」と断定するのではなく、「この書き方だとメモリリークが発生する可能性があるかもしれません。○○のようにするのはいかがでしょうか」と提案する形が一般的だ。このような書き方は、表面上は遠回しに見えるが、相手への敬意を示しながら問題を的確に伝えるための日本語特有の表現技法である。

また、コメントには「nits:」や「提案:」「質問:」といったプレフィックスをつける慣習も広まっている。これにより、指摘の深刻度や意図が明確になり、レビュイーが優先順位をつけて対応しやすくなる。コードレビューを円滑に進めるためには、技術的な知識と同時に、こうした日本語のコミュニケーションスタイルへの理解が欠かせない。

## ❓ 読解問題

1. 日本のコードレビューでは、なぜ直接的な否定表現を避けるのですか？
2. 「nits:」「提案:」などのプレフィックスをつける目的は何ですか？
3. コードレビューを円滑に進めるために必要なこととして、本文が挙げているものを2つ答えてください。

<details>
<summary>解答</summary>

1. 受け取る側が萎縮しないようにするため。
2. 指摘の深刻度や意図を明確にし、レビュイーが優先順位をつけて対応しやすくするため。
3. 技術的な知識と、日本語のコミュニケーションスタイルへの理解。

</details>

## 📐 文法ポイント

### 1. 〜ようにする（N3）
> 受け取る側が萎縮しない**ようにする**

「〜ようにする」= "to make it so that ~" / "to try to ~"（意図的な配慮）  
例：ミスが起きないようにチェックリストを使う。

### 2. 〜のではなく〜（N3）
> 「間違っています」と断定する**のではなく**、提案する形が一般的だ

「〜のではなく〜」= "not ~ but rather ~"（対比・訂正）  
例：報告するのではなく、まず相談するべきだった。

### 3. 〜と同時に（N2）
> 技術的な知識**と同時に**、コミュニケーションスタイルへの理解が欠かせない

「〜と同時に」= "at the same time as ~" / "as well as ~"（並列・同時性）  
例：機能を追加すると同時に、テストも書く必要がある。
